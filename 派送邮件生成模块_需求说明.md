# 派送邮件生成器 · 开发需求说明

> 目标：在现有 FastAPI + Jinja2 项目里**新增一个独立模块**，与 `做账单 / 香港进口清关 / 香港出口清关 / 财务记录 / UFO 邮件生成器` **同级并列**（一起出现在首页 `index.html` 的模块卡片里）。
>
> 输入：「客户发来的原始 .eml 邮件」
> 输出：一封可直接发给仓库的「派送邮件 .eml」
> 中间帮我做：附件解包改名、按 Tan# 拆段、自动截总表区域、生成正文模板等所有重复劳动。

---

## 一、模块定位

**这是一个全新的独立模块，不要修改任何现有模块的代码逻辑。** 与 UFO 邮件生成器是平级的兄弟模块，只是恰好都做"邮件 .eml"。

需要新增的文件（仅供命名参考，可调整）：

| 类型 | 路径 | 作用 |
| --- | --- | --- |
| 新增模板 | `templates/dispatch_mail.html`（Step 1 上传/预览） | 主入口页 |
| 新增模板 | `templates/dispatch_mail_compose.html`（Step 2 填邮件信息） | 第二步页面 |
| 新增模块 | `dispatch_mail_store.py` | 解析、配对、改名、生成 .eml 的核心逻辑 |
| 新增模块 | `dispatch_mail_image.py`（可选，截图算法独立成文件） | 总表区域截图 |
| 新增 sqlite 表 | `dispatch_mail_settings`（如做"记住常用收件人/仓库地址"） | 持久化 |

**只允许动这两个现有文件**（且只是追加，不改既有逻辑）：

- `reconcile_web_app.py`：追加 6 个新路由 + 1 个 `init_dispatch_db()` 调用

**严禁改动**：`ufo_mail_store.py`、`templates/ufo_mail.html`、其他已有模块的任何代码。

样式可以复用 `static/styles.css` 里既有的通用类（`.shell.wide / .panel / .topbar / .primary-btn / .secondary-btn / .field / .status-chip` 等），新增样式请用专属前缀 `.dispatch-xxx` 追加到文件末尾，**不要修改现有 CSS 规则**。

> 写法风格请参考 UFO 邮件生成器（`ufo_mail_store.py` + `templates/ufo_mail.html`）——但这是"参考已有代码风格"，**不是"把代码加到那里"**。

## 二、业务背景

客户每周让我们去他们工厂装货，事后通过邮件发来本批货物的资料：

- **Tan# 总表**（Excel，例如 `APR.15.xls`）：本批所有货物的总表，按 Tan# 分票
- **单页 PL & INV**（Excel，命名固定为 `DQTH + 数字`，例如 `DQTH01.xls`）：每票一份，含装箱单和发票
- **SO 交仓文件**（任意格式，PDF / Excel / Word 都可能）：每票一份，命名不规范

我们要做的事：
1. 从客户邮件里把附件全部取出
2. 按规则把 DQTH 和 SO 文件**改名**
3. 按总表的 Tan# 顺序，逐票生成派送邮件正文段落
4. 把改名后的所有附件 + 总表打包，连同正文，生成一封 .eml 发给仓库

## 三、输入物

### 3.1 客户邮件 `.eml`
样本：`C:\Users\ac\Desktop\sample from customer.eml`

里面附件可能包含：
- 1 份 Tan# 总表（也可能漏）
- N 份 DQTH 单页 PL & INV（也可能漏一两份）
- N 份 SO 交仓文件（也可能漏，且文件名不规范）

### 3.2 用户在 GUI 上手填
- 运输单号
- 预计到仓日期 Y 号
- 预计到仓时间 N 点
- 卸货车牌 AAA
- 每一票的交仓地址（程序不要试图自动猜，留空让用户填）
- 每一票的截仓注意事项备注（默认值「截仓一般为隔天17点」可以预填，让用户改）

## 四、输出物

### 4.1 一封派送邮件 `.eml`
样本：`C:\Users\ac\Desktop\sample to warehouse.eml`

#### 邮件主题
```
{运输单号} 伟创力出口{总板数}板，预计{Y}号{N}点前到倉庫 貨物到倉庫有扁箱請第一時間拍照通知，謝謝
```
> 注意：是繁体「倉庫 / 請」，跟客户既有习惯保持一致。

#### 邮件附件
- 客户原始的 Tan# 总表（不改名，原样附上）
- 所有改名后的 DQTH 单页（命名见 §6.1）
- 所有改名后的 SO 交仓文件（命名见 §6.2）

#### 邮件正文（HTML）
```
Dear Kinki

请查收附件的交仓文件和装箱单，预计{N}点前到仓库，卸货车牌 {AAA}，
到仓后卸货请注意，有破损请第一时间通知，
另外请注意交仓要求和时间，谢谢。
（注意伟创力的发票不可拿去交仓，装箱单发票仅供仓库核对货）

一：{第1票卡板数}板交{仓库代号}仓：{地址 - 用户填}

[嵌入图片：从总表中截取的"该票区域"]

注意事项：{截仓一般为隔天的17点，是否需要寄尾纸等 - 用户填/可改}

----------------

二：{第2票卡板数}板交{仓库代号}仓：{地址 - 用户填}

[嵌入图片：从总表中截取的"该票区域"]

注意事项：...

----------------

...以此类推
```

> 「该票在总表中对应的图片」**不是指 Excel 内嵌图片**，而是指**对总表的截图**：
> - 范围起点：该票第一行（即该 Tan# 的序号所在行）
> - 纵向边界：向下直到遇到下一个 Tan# 那一行的上一行（不包含下一票）
> - 横向边界：向右一直到「卡板数」这一列（包含该列）
> - 截图建议输出 PNG，通过 `cid:` 嵌入 HTML 邮件正文

## 五、数据结构与关联规则

### 5.1 Tan# 总表
样本：`C:\Users\ac\Desktop\APR.15 .xls`

关键列：`Tan#`、`Customer PO`、`毛重KG`、`箱数`、`卡板数`、`备注`

**重要规则**：
- **一个 Tan# 可能跨多行**（同一票拆成几个 PO 或几行明细）
- 同一 Tan# 的「箱数 / 毛重 / 卡板数」需求和聚合
- 「备注」列文字会跟 SO 交仓文件名相似，可用于配对

### 5.2 DQTH 单页 PL & INV
样本：`C:\Users\ac\Desktop\DQT01.xls`

- 命名规律：`DQTH + 数字`（例：`DQTH01.xls`、`DQTH02.xls`）
- 不带 Tan# 字段
- 关键列：`订单号 Customer PO`、`箱数 CTN`、`卡板 Pallet`、`总毛重 Gross weight(kgs)`
- **一份 DQTH 内部也可能多行，需汇总**

### 5.3 关联规则

#### DQTH ↔ Tan# 配对（必须能自动跑通）
1. **正常情况**：客户邮件里 DQTH 数量 = 总表 Tan# 数量（一一对应）
2. **第一步**：用 `Customer PO` 配对
3. **第二步**：当多个 Tan# 共享相同 PO 时，用「箱数 / 毛重 / 卡板数」三个数字交叉校验，谁全对就是谁
4. **第三步**：仍配不上则标红，列在「未匹配区」让用户手动指认

#### SO 交仓文件 ↔ Tan# 配对（容忍失败）
- 用 SO 文件名 vs 总表「备注」列文字的**相似度**判断（推荐 `rapidfuzz` 或 `difflib.SequenceMatcher`，阈值可配）
- **不强求 100% 匹配**：匹配不上的 SO 文件保留原名，列在「未匹配区」让用户手动拖拽指认
- 用户也可以选择「无 SO 文件」（漏发的情况）

## 六、附件改名规则

### 6.1 DQTH 单页改名
- 命名格式：`{N}板装箱单.xls`（N = 该 Tan# 汇总后的卡板数）
- 多个 Tan# 卡板数相同时按总表顺序加序号：`{N}板装箱单(1).xls`、`{N}板装箱单(2).xls`、`{N}板装箱单(3).xls`...
- 例：总表里 3 票分别是 5 板 / 5 板 / 8 板，改名为 `5板装箱单(1).xls`、`5板装箱单(2).xls`、`8板装箱单.xls`
- 扩展名沿用原文件

### 6.2 SO 交仓文件改名
- 命名格式：`{N}板交仓文件.{原扩展名}`
- 多票同板数同样用 `(1) (2)` 顺延
- 匹配不上的保留原名，**不要瞎改**

### 6.3 总表
- 不改名，原样作为附件加入

## 七、GUI 流程（两步走）

### Step 1：解析 + 改名预览（不需要地址等下游信息）

1. 上传区：拖入或选择客户原始 `.eml`
2. 后端解析后，页面展示一张「派送票预览表」，每行一票（按总表 Tan# 顺序）：
   | 序号 | Tan# | 板数 | 箱数 | 毛重 | Customer PO | 配对的 DQTH（原名 → 改名后） | 配对的 SO 交仓（原名 → 改名后） | 状态 |
3. 下方两个「未匹配区」：
   - 未匹配的 DQTH 文件（少见）
   - 未匹配的 SO 文件（常见）
   - 用户可拖拽 / 下拉指派给某一票，也可以勾选「忽略」
4. 用户可以**手动覆盖任何一行的改名结果**（输入框直接改）
5. 顶部显示警告横幅（如果有）：
   - 总表缺失 / DQTH 数量与 Tan# 数量不一致 / 某票未配到 DQTH 等
6. 按钮：`下一步 · 填邮件信息`

### Step 2：填邮件信息 + 生成 eml

1. 全局信息表单（一组输入）：
   - 运输单号
   - 预计到仓日期（Y 号）
   - 预计到仓时间（N 点）
   - 卸货车牌（AAA）
   - 收件人 To / 抄送 Cc / 发件人 From（建议用 sqlite 记住上次值，参考现有 `ufo_mail_settings`）
2. 每票一组输入（按 Step 1 的票顺序展开）：
   - 仓库代号（如 G）
   - 交仓地址（多行文本）
   - 注意事项（默认预填「截仓一般为隔天的17点」，可改）
3. 右侧实时预览：邮件主题 + 邮件正文（HTML）
   - 截图直接渲染出来给用户看效果
4. 按钮：`生成 .eml 并下载`

> 之所以拆两步：第一步只依赖 .eml 本身，可以独立完成「文件改名 + 总表截图」，让用户先把附件搞定；第二步再补地址等"业务信息"。

## 八、边界与异常处理

| 情况 | 行为 |
| --- | --- |
| 客户邮件没附件 | Step 1 直接报错，提示重新上传 |
| 缺总表 | Step 1 红色横幅提醒，仍允许进入 Step 2，但「票拆段」会退化成空白模板让用户全手填 |
| DQTH 数量 ≠ Tan# 数量 | 黄色横幅提醒，缺的那票在表里标红，DQTH 列空着，让用户决定继续还是放弃 |
| SO 文件全部对不上备注 | 不报错，全部进「未匹配区」让用户手动指认 |
| 一个 Tan# 在总表里跨多行 | 自动汇总「箱数 / 毛重 / 卡板数」，配对用汇总值 |
| 一份 DQTH 内部多行 | 同上，汇总后再配对 |
| 用户改名造成文件名冲突 | 自动加 `(2) (3)` 后缀，并在 UI 上提示 |
| 用户没填地址 | Step 2 不阻塞生成，但生成的邮件该位置留空（方便用户在 Outlook 里再补） |

## 九、技术建议

- Web 框架：FastAPI + Jinja2（与现有项目一致，**不引入** React / Vue）
- 邮件：标准库 `email.message.EmailMessage` + `email.policy.SMTP`
- 解析 .eml：标准库 `email.parser.BytesParser`
- 读取 xls / xlsx：`xlrd==1.2.0`（旧 .xls）+ `openpyxl`（.xlsx）；如果不挑剔可统一用 `pandas`
- 字符串相似度：`rapidfuzz`（性能比 `difflib` 好）
- 总表区域截图（关键难点）：
  - 思路 A：用 `openpyxl` 读取目标区域 → 用 `Pillow` 自己渲染成表格 PNG（可控、依赖少、推荐）
  - 思路 B：用 `xlsx2html` + `imgkit/wkhtmltoimage` 转图片（依赖系统二进制，部署麻烦，不推荐）
  - 思路 C：调 LibreOffice headless 转 PDF 再切页（重，最后兜底）
- HTML 邮件嵌图：`message.add_related(img_bytes, 'image', 'png', cid='ticket_1')`，正文 HTML 用 `<img src="cid:ticket_1">`

## 十一、新模块的项目内位置

**只新增、不改写**。该模块和 UFO 邮件、做账单、清关等现有模块完全独立。

### 11.1 在 `reconcile_web_app.py` 追加（不动既有路由）
- `GET  /modules/dispatch-mail` → Step 1 上传页
- `POST /modules/dispatch-mail/parse` → 解析 .eml 后重定向到预览

- `GET  /modules/dispatch-mail/compose/{session_id}` → Step 2 填邮件信息
- `POST /modules/dispatch-mail/generate/{session_id}` → 生成 .eml 并下载
- 顶部 `import` 一份新模块 `dispatch_mail_store`
- 启动时调用一次 `init_dispatch_db()`（如果用了 sqlite）

### 11.2 在 `templates/index.html` 追加 1 张模块卡
- 紧跟现有 5 张卡之后，作为第 6 张
- 标题：`派送邮件生成器`
- 副标题：`客户邮件 → 仓库派送邮件`
- 描述：`从客户邮件提取附件，自动改名、按 Tan# 拆段、生成发给仓库的 .eml。`
- 链接：`/modules/dispatch-mail`
- 图标字母：`D`、徽章：`派送邮件`
- 视觉沿用现有风格（米黄 + 橙棕 + 绿），不要引入新配色

### 11.3 复用既有目录与机制
- session 数据放内存即可（用与现有相同的 `SESSION_STORE: Dict[str, Dict[str, Any]]`，可在 store 模块里另起一个独立 dict 也行）

- 生成的 .eml 存 `runtime/outputs/`
- 截图临时图片建议存 `runtime/outputs/dispatch_{session_id}/ticket_{i}.png`
- 持久化（如常用收件人）用 sqlite，文件名建议 `runtime/dispatch_mail.db`，表名 `dispatch_mail_settings`
