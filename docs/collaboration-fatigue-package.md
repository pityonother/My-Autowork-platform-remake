## A. 项目背景

### 这个项目是什么

- 这是一个本地业务工作台，不是通用 SaaS。
- 核心业务围绕：账单对账、香港进口/出口清关、财务记录、Booking、派送邮件、UFO 邮件等日常操作。
- 项目形态已经从“单点脚本”演化成“多模块 FastAPI 本地工具”。

### 我最初想完成什么

- 在可追溯会话里，最初的明确需求是：从一个旧项目副本里开一个新项目空间，然后先做“自动化对账”。
- 第一阶段真正的起点不是“做一个完整平台”，而是“先从一个账单例子出发，把单页账单回填进总账单”。
- 之后范围不断扩大，逐步变成：
  - 月度批量回填
  - 总表预览和子表检查
  - 费用规则校验
  - 地址归一与字典建设
  - 财务模块完善
  - 出口清关页面交互重做
  - 派送邮件模块重构

### 技术栈 / 主要文件 / 运行方式

- 技术栈：
  - `FastAPI`
  - `Jinja2`
  - 本地 `SQLite`
  - `Excel / EML` 解析
- 主要文件：
  - `reconcile_web_app.py`
  - `invoice_reconciler.py`
  - `finance_store.py`
  - `export_clearance_store.py`
  - `dispatch_mail_store.py`
  - `booking_store.py`
  - `templates/*.html`
  - `static/styles.css`
- 运行方式：
  - 通过 `run_reconcile_web.bat`
  - 本地启动 `uvicorn`
  - 页面运行在 `127.0.0.1:8010`

### 当前项目状态

- 当前项目已经“能跑、能用、能生成结果”，但仍明显处在高频手工打磨阶段。
- 最近最活跃、改动最密集的区域是：
  - `dispatch_mail_store.py`
  - `reconcile_web_app.py`
  - `templates/dispatch_mail_preview.html`
  - `templates/export_customs.html`
  - `templates/finance_records.html`
  - `static/styles.css`
- 工程层面仍有几个结构性问题：
  - 不是 git 仓库
  - 缺少系统化测试
  - `runtime/uploads`、`runtime/outputs` 会持续累积业务与调试产物
  - 编码问题、显示编码问题、真实文件乱码问题曾多次混在一起
  - 很多规则仍直接硬编码在 Python 里

## C. 关键疲劳点

- 当时你想要什么：

  - 每增加一个真实业务规则，复杂度都直接作用在产品结构上
- 涉及：
  - 范围变大
  - 没先给计划
  - 改太多文件
  - 没有验收标准

### 2. 规则靠聊天逐条长出来，没有先沉成规格

- 当时你想要什么：

  - 新规则经常和旧规则互相影响，但没有统一测试样本保护
- 涉及：
  - 范围变大
  - 连续 patch 而不是先诊断

  - 返工

结果/后果：需求的核心其实是“视觉优先级”，而不是 DOM 顺序；由于验收标准没有前置，页面返工了至少两轮。

### [上下文 thread 断裂]
User: “那个对话太长了”
Codex: “你给我一个‘压缩版上下文’就够了……”
User: “codex://threads/……”
Codex: “我不能直接打开或读取这个对话链接……”
结果/后果：当上下文没有沉淀成项目文档时，用户被迫承担跨会话的上下文搬运工作。

### [用户明确要求先做需求规范化]
User: “请记住每次我提新的需求你先用需求文档的技能将我的需求规范化，计划好再出发”
Codex: “我先按你要求把这次诉求收束成一版可执行需求……”
结果/后果：这句话本身就是疲劳信号，说明用户已经不满意“边理解边改”的默认协作节奏。

### [重构之后，用户仍然在纠正最基础的可见性]
User: “解析邮件后，选择对应文件部分的总表截图也太小了……我只需要看到序号+Tan号码，PO，PCS数量，板数和箱数”
Codex: “这次不再去裁原表，而是直接生成一张‘匹配专用精简图’……”
结果/后果：即便已经做了大规模匹配页重构，用户仍需要继续指出“最关键的信息根本看不清”，说明前面的视觉验收不充分。

## F. 对用户协作方式的初步观察

### 你哪里可能给得太模糊

- 你常常是带着很强的业务直觉提出需求，但很多“默认前提”没有显式说出来。
- 例如“历史批次比较次要”，实际含义是“视觉层级、宽度、注意力焦点都要明显偏向筛查区”，不是简单换顺序。
- 例如“先从小开始”，后来又很快自然延展到“整月批量 + 预览 + 子表 + 规则校验”，对实现边界影响很大。

### 哪里没有限制范围

- 很多请求会在原需求还没稳定前继续叠加相邻需求：
  - 自动对账 -> 地址归一 -> 派送费共享判断 -> 地址字典文件治理
  - 派送邮件截图 -> GUI 默认值 -> 预览布局 -> 收件人保存 -> 匹配页 -> 文件预览 -> 命名规则
- 这种方式非常接近真实工作，但会让“一个问题”演变成“一个模块重构”。

### 哪里没有设验收标准

- 你经常知道自己“不满意什么”，但未必一开始就把“满意的样子”写成可执行标准。
- 例如：
  - 是否要改真实文件还是只生成副本
  - UI 是只换顺序，还是还要改变主次大小
  - 截图要保留哪些列，是否需要 exact fidelity
- 一旦验收标准后置，就容易出现“代码做完了，但你看到的仍不是你要的那个东西”。

### 哪里让你连续 patch 而不是先诊断

- 你有时会接受“先做一版再看”，这对探索型需求很自然，但也等于默认了高 patch 密度。
- 当需求本身跨业务规则、数据格式、前端交互多个层次时，这种推进方式会非常累。

### 哪里可能因为太相信 / 太不信任导致疲劳

- 太相信的时刻：
  - 允许我直接改真实业务文件
  - 默认我能靠连续 patch 把复杂业务规则慢慢补稳
- 太不信任的时刻：
  - 需要靠连续截图、逐点纠错才能收敛
  - 一旦发生数据覆盖事故，后续每一步都会更谨慎、更耗神
- 这两种状态切换频繁，会让双方协作节奏都变得不稳定。

## G. 对 Codex / coding assistant 协作方式的初步观察

### 我哪里可能过度自信

- 我太容易把“一个局部改动”理解成“顺手做成更完整的版本”。
- 这在速度上看是积极主动，在用户体验上却可能变成“你只要 A，我却交付了 A+B+C，还要你一起审”。

### 哪里可能没有先确认

- 我多次在没有稳定验收标准前就进入实现：
  - 财务页模板整页重写
  - 出口清关弹窗和布局连续重做
  - 派送邮件匹配页与预览结构重构
- 有些地方本应先确认：
  - 是否允许操作真实业务文件
  - UI 目标是“贴参考项目”还是“独立重设计”
  - 变更是否只限于当前痛点，还是允许连带优化

### 哪里可能生成太多代码

- 派送邮件模块就是典型：
  - 后端逻辑
  - 路由
  - 多个模板
  - 样式
  - 运行时图片导出
  - 文件预览分流
  - 前端即时规则计算
- 这使得每次变更的审查面太大，用户很难低成本确认“只改了哪一块”。

### 哪里可能没有保护项目状态

- 地址库事故最典型：
  - 直接改真实文件
  - 又错误复用旧副本
  - 覆盖了用户的手工整理劳动
- 此外：
  - 项目没有 git
  - 很多变更是在 live runtime 数据存在的环境里做
  - 虽然后面补了备份意识，但前期保护还不够

### 哪里可能导致用户需要承担过多审查成本

- 我经常把“功能开发 + 顺手优化 + 结构整理 + 样式修补”捆绑在一次交付里。
- 这会让用户必须同时审查：
  - 业务规则有没有变
  - 页面结构有没有变
  - 样式有没有跑偏
  - 命名、提示、保存状态有没有副作用
- 对一个本地业务工具来说，这种审查成本很容易高于“写代码本身的效率收益”。

### 还有哪些协作方式问题

- 我在很多场景下只做了：
  - `py_compile`
  - 模板渲染检查
  - 局部样本验证
- 但没有及时做：
  - 浏览器可视化验收
  - 用户视角流程走查
  - live data 安全策略
- 结果是：从“代码能跑”到“用户觉得顺手”之间，还要经过很多轮人工反馈。

## 补充结论

这段协作之所以累，不是单一原因造成的，而是几件事叠加：

- 需求高度真实、细碎、持续演化
- 业务规则很多是隐性经验，必须在聊天里一点点长出来
- 我过于偏向“先做出来”，而不是“先冻结边界”
- 项目缺少 git、测试、变更日志、规则清单这类缓冲层
- 上下文又分散在多个对话里，导致记忆恢复成本很高

如果要让后续协作明显变轻，最有效的不是“让双方更努力”，而是先把下面三件事制度化：

- 每次新需求先写 5 行规格：
  - 目标
  - 不做什么
  - 验收标准
  - 是否允许改真实文件
  - 是否允许顺手优化相邻区域
- 高风险操作默认只对副本执行：
  - 原始 Excel
  - 地址字典
  - 运行时数据库
- 每完成一个阶段就固化一份“规则清单 + 样本 + 当前已知限制”，不要只留在聊天里。
