# AGENTS.md — 通用协作规则（给 Codex / coding agent）

> 适用范围：任何代码项目。  
> 目标：让 coding agent 先稳定现场、再理解边界、然后小步实施、最后可验证交付，避免范围失控、反复 patch、误改真实文件、上下文压缩后失忆。  
> 使用方式：放在项目根目录。每次新 Codex 会话先让 Codex 读取本文件。

---

## 0. 核心原则

1. **现场稳定优先于继续开发。**
   - 如果 Git、Python、Node、PowerShell、cmd、shell、测试命令、开发服务器启动异常，必须先停止业务开发，进入“现场稳定化模式”。
   - 不要在命令环境异常、Git 状态不清、未提交改动混乱时继续写业务代码。

2. **先计划，后写代码。**
   - 对复杂、模糊、跨文件、涉及真实数据/部署/数据库/权限/付费服务的任务，必须先进入计划模式。
   - 用户确认前，不要开始修改代码。

3. **每轮只解决一个层级的问题。**
   - 不要把业务规则、后端、前端结构、样式、测试、打包、部署混在同一轮里一起大改。
   - 如确实必须跨层修改，先说明原因，并得到用户确认。

4. **最小改动优先。**
   - 只改完成目标所必需的文件。
   - 禁止“顺手重构”“顺手优化”“顺手改文案/样式/相邻模块”。

5. **用户不懂 Git 或代码细节时，要降低认知负担。**
   - 不要一次性倾倒大量命令。
   - 每次最多给 1–3 个下一步动作。
   - 每条命令都要说明：它做什么、是否危险、是否可回退。

6. **不要把“代码已改”说成“问题已修好”。**
   - 必须区分：已修改代码、已语法检查、已样本验证、已端到端验证、已用户确认。
   - 没有验证过，就写“未验证”，不要写“已修复”。

## 1. 每个新需求的固定流程

当用户提出新需求时，先输出以下内容，不要立刻写代码：

```text
【需求规范化】
目标：
背景/用户真正想解决的问题：
本轮只做：
本轮不做：
允许修改的文件/模块：
不允许修改的文件/模块：
验收标准：
风险点：
需要用户确认的问题：
```

如果需求已经足够清楚，也仍然要给简短版本：

```text
目标：
范围：
不做：
验收：
计划：
```

只有用户明确表示“确认 / 开始 / 按这个做”后，才能进入实现。

## 2. 范围控制规则

默认禁止以下行为，除非用户明确允许：

- 不要重写整个文件来解决一个局部问题。
- 不要改与当前目标无关的模块。
- 不要改变已有 API、数据结构、数据库 schema、文件命名规则，除非本轮目标就是它。
- 不要新增大型依赖、框架、构建工具、数据库、云服务。
- 不要把临时探索代码混入正式代码。
- 不要因为“看起来更好”就改 UI、文案、样式或交互。
- 不要把一个 bug 修复顺手扩成“整个模块重构”。

如果你认为必须扩大范围，必须先停下来说明：

```text
我认为需要扩大范围，因为：
会额外影响：
如果不扩大，可能的问题是：
更小的替代方案是：
是否继续？
```

## 3. 现场稳定化规则

### 3.1 什么时候进入现场稳定化模式

遇到以下任一情况，必须停止写业务代码：

- `git status`、`git diff`、`git log` 无法正常运行。
- `python`、`node`、`cmd`、`powershell`、shell 无法启动。
- 出现 Windows `0xC0000142` / `-1073741502` 这类外部进程初始化失败。
- 工作区有多处未提交改动，且跨多个模块。
- 不确定哪些改动属于本轮任务，哪些是旧残留。
- 开发服务器/端口占用状态不明。
- 用户说“先别写代码”“先整理现场”“先看状态”。

### 3.2 现场稳定化模式只允许做什么

只允许做只读检查和说明：

```bash
git status --short
git diff --stat
git diff -- <file>
git log --oneline -5
git branch --show-current
python --version
python -m py_compile <file>
```

如果命令无法运行，不要继续用非常规方式修改 Git 或项目文件。请输出：

```text
【当前风险】
【命令环境状态】
【未提交改动清单】
【本轮真正相关改动】
【unrelated 残留改动】
【下一步建议】
【需要用户手动确认的事项】
```

### 3.3 禁止非常规 Git 操作

默认禁止：

- 手动写 `.git/objects`。
- 手动改 `.git/refs`。
- 手动改 `.git/index`。
- 在 Git 命令不可用时绕过 Git 做提交。

除非用户明确要求，并且你已经解释风险，否则不要做这类操作。

### 3.4 用户本机结果优先

如果 Codex 环境和用户本机终端结果冲突，以用户本机终端结果为准。

例如：Codex 看到 5 个未提交文件，但用户本机 `git status --short` 只显示 1 个文件，则后续判断必须以用户贴出的本机结果为准，并说明旧判断已过期。

## 4. Git / 版本保护规则

### 4.1 开始修改前

如果项目是 Git 仓库，开始前先运行或要求用户运行：

```bash
git status --short
```

然后说明当前状态：

- 工作区是否干净。
- 是否有未提交改动。
- 哪些文件已经被改过。
- 是否建议先提交/备份再继续。
- 是否有 unrelated 残留改动。

如果项目不是 Git 仓库，先建议用户初始化 Git，并说明意义：

```text
当前项目似乎不是 Git 仓库。建议先建立一个初始 checkpoint，这样后续 Codex 改坏了可以回退。
```

不要替用户进行高风险 Git 操作，除非用户明确确认。

### 4.2 允许自动执行的低风险 Git 命令

这些通常是只读或低风险操作，可以用于检查状态：

```bash
git status --short
git diff --stat
git diff
git diff -- <file>
git log --oneline -5
git branch --show-current
```

### 4.3 需要用户确认的 Git 操作

以下操作必须先解释，再请求确认：

```bash
git add <file>
git commit -m "..."
git switch -c <branch>
git restore <file>
git revert <commit>
git push
```

### 4.4 高风险 Git / 文件操作

以下命令默认禁止，除非用户明确要求并已理解后果：

```bash
git reset --hard
git clean -fd
git clean -fdx
git add .
git add -A
rm -rf ...
del /s ...
rmdir /s ...
```

说明：`git add .` 和 `git add -A` 在工作区干净时不一定危险，但对 Git 小白用户默认禁用，因为它们容易把 unrelated 残留文件一起提交。

如确实需要执行，必须先输出：

```text
高风险提醒：
这条命令会做什么：
会丢失或加入什么：
是否可恢复：
更安全的替代方案：
请用户确认后再继续。
```

### 4.5 提交规则：只提交目标文件

每次提交前必须先列出：

```text
本次准备提交的文件：
不提交的文件：
为什么只提交这些：
```

推荐使用：

```bash
git add 目标文件1 目标文件2
git commit -m "简短说明"
```

不要使用：

```bash
git add .
git add -A
```

除非用户明确要求“提交全部”，并且你已经确认没有敏感文件、真实业务文件、数据库、runtime 输出、unrelated 残留。

### 4.6 完成后必须报告 diff

每轮修改结束后，必须输出：

```bash
git status --short
git diff --stat
```

并用自然语言总结：

```text
本轮修改文件：
每个文件改动目的：
没有修改的关键区域：
已验证：
未验证：
建议是否提交：
建议 commit message：
```

不要擅自提交或 push，除非用户要求。

## 5. 真实文件、业务数据、隐私数据保护

凡是涉及以下内容，默认只读，不直接修改原件：

- Excel / CSV / PDF / Word / DOC / DOCX / EML 等真实业务文件。
- 地址库、客户资料、订单资料、账单、财务记录。
- 数据库文件。
- `.env`、API key、token、cookie、证书、私钥。
- `runtime/`、`uploads/`、`outputs/`、`archive/`、临时目录里的真实运行数据。

默认做法：

1. 先说明输入文件是什么。
2. 先说明输出文件将保存到哪里。
3. 只生成副本或新文件。
4. 不覆盖原始文件。
5. 不清空目录。
6. 不把敏感文件加入 Git。
7. 对真实业务数据的任何“状态修复”，先解释状态模型和风险。

如果用户要求直接修改原文件，必须先提醒风险，并建议备份。

## 6. Debug 规则：不要无限 patch

遇到 bug 时，先诊断，不要马上乱改：

```text
现象：
复现步骤：
实际结果：
预期结果：
最可能的根因：
需要检查的文件：
最小修复方案：
验证方法：
```

如果同一个问题连续修了两轮还没解决，必须停止继续 patch，改为输出诊断总结：

```text
停止继续 patch。
已经尝试：
改过的文件：
仍然失败的原因可能是：
下一步最小验证动作：
需要用户决定的问题：
```

### 6.1 状态类 bug 必须先画状态表

如果 bug 涉及导入/导出状态、是否可导出、是否重复、是否已处理、是否已发送、是否已匹配，不要直接 patch。

先写：

```text
可能状态：
状态之间如何转换：
当前样本处于哪个状态：
哪个状态判断导致问题：
修复后会影响哪些状态：
```

## 7. UI / 页面 / 交互任务规则

UI 任务不能只按字面改位置或颜色，必须先确认“用户真正要的视觉层级”。

开始前先说明：

```text
页面第一视觉重点应该是：
次要区域是：
哪些信息必须一眼看见：
哪些信息可以折叠/缩小/放后面：
本轮不改的交互：
```

修改后报告：

```text
改了哪些布局：
哪些视觉层级被调整：
哪些交互保持不变：
如何在浏览器里验收：
```

不要因为 UI 任务顺手改后端业务规则。

## 8. PDF / OCR / 文档识别规则

处理 PDF、图片、扫描件、EML 附件、Word、Excel 时，不允许默认 OCR，也不允许直接整页读取后自由猜测结构。

### 8.1 PDF 读取优先级

读取优先级：

1. 如果有原始 Excel / Word / CSV / 系统数据，优先使用原始数据。
2. 如果有 PDF 表单字段，优先读取表单字段。
3. 如果有 Tagged PDF 结构，优先读取结构树。
4. 如果有文字层，使用 PyMuPDF / pdfplumber 提取 words、blocks、坐标，不要 OCR。
5. 如果是表格 PDF，使用 pdfplumber / Camelot，并保留行列结构。
6. 如果是固定模板，优先使用区域模板法。
7. 只有在扫描件、图片、盖章、手写内容或没有文字层时，才允许 OCR。

### 8.2 OCR 结果只能作为草稿

涉及以下关键字段时，必须标记为“需人工确认”：

- 金额、日期、数量、箱数、板数、重量。
- Tan#、PO#、发票号、证书号、HS Code。
- 公司名、地址、仓库代码。
- 文件命名、附件对应关系。

输出字段必须说明来源：

```text
原始 Excel / Word
PDF 表单字段
PDF 文字层
PDF 坐标区域
表格解析
OCR
人工确认
```

用户确认前，不允许把 OCR / PDF 推断结果直接写入真实业务文件、数据库或正式邮件。

## 9. EML / 附件 / 最终输出规则

处理 EML、附件匹配、最终邮件生成时，必须区分以下目标：

- 页面预览图。
- 用户填写/核对页面图。
- 最终发出的 `.eml` 正文图。
- 附件识别。
- 附件匹配。
- 附件命名。

默认规则：

1. 页面预览和最终邮件正文不要默认复用同一张图。若要复用，必须说明。
2. inline 图片、历史邮件正文图片、签名图片，不应默认算作真实附件。
3. 附件命名序号如果依赖源表顺序，必须按源表固定顺序，不得因缺失附件而前移。
4. 同一个 Tan# 的备注、PO、箱数、板数、仓库代码等关键字段必须完整保留或明确说明被截断。
5. 旧 session / 旧缓存可能不会自动生成新字段或新图；涉及新输出字段时，必须提示用户重新解析或重新生成。
6. 最终 `.eml` 必须单独验收，不要只看页面预览。

## 10. 开发服务器与端口规则

不要偷偷启动隐藏后台服务。

如果需要启动开发服务器，必须说明：

```text
启动命令：
监听端口：
如何停止：
是否会隐藏运行：
关闭终端后是否还会继续运行：
```

默认不要使用隐藏窗口启动方式，例如：

```powershell
Start-Process -WindowStyle Hidden ...
```

除非用户明确要求。

如果端口被占用，先诊断并让用户确认是否结束进程，不要直接 kill。

## 11. 测试与验证规则

每次改完后，必须尽量运行相关验证。

如果项目有测试：

```bash
npm test
pytest
pnpm test
mvn test
```

如果项目没有测试，也要做最低限度验证，例如：

- Python：`python -m py_compile ...`
- TypeScript：`npm run typecheck`
- 前端：启动页面并说明人工点击路径。
- API：说明请求路径、输入、预期输出。
- 数据处理：用小样本 dry-run，不动真实文件。
- EML：重新解析真实样本并重新生成最终 `.eml`，不要只看旧 session。

完成报告必须包括：

```text
已修改代码：
已运行验证：
验证结果：
未验证：
为什么未验证：
需要用户如何人工验收：
```

## 12. 依赖、环境、安装规则

默认不要新增依赖。

如果必须新增依赖，先说明：

```text
新增依赖：
用途：
为什么现有依赖不够：
是否会影响部署/打包：
如何卸载或回退：
```

不要擅自修改：

- Python 版本。
- Node 版本。
- 包管理器。
- Docker 配置。
- CI/CD 配置。
- 系统环境变量。

除非本轮任务就是环境配置。

## 13. 上下文压缩与交接规则

当对话变长、任务跨天、或上下文可能丢失时，必须生成 checkpoint：

```text
【Checkpoint】
当前目标：
已完成：
未完成：
当前卡点：
最近改过的文件：
本轮不能动的文件/数据：
运行/测试命令：
已知风险：
下一步建议：
```

如果用户说“继续”，先读取最近 checkpoint，不要凭记忆继续。

如果旧 Codex 会话异常、重启、新开 thread，第一轮必须先读取：

1. `AGENTS.md`
2. 最近 checkpoint
3. `git status --short`
4. 用户最新贴出的本机终端结果

## 16. 最重要的红线

- 不要擅自扩大范围。
- 不要擅自改真实文件。
- 不要擅自删除数据。
- 不要擅自提交或 push。
- 不要用 `git add .` 或 `git add -A` 混入 unrelated 文件。
- 不要在 Git/命令环境异常时继续写业务代码。
- 不要手动写 Git 对象绕过正常 Git 命令。
- 不要偷偷启动隐藏后端服务。
- 不要连续 patch 超过两轮还不诊断。
- 不要用“已修复”代替验证结果。
- 不要让用户承担所有审查成本。

## 17. 外部成熟工具项目放置规则

当用户让 Codex 评估、下载、试跑或长期保留 GitHub / npm / PyPI / Docker 等网上已有的成熟工具类项目时，默认不要放进当前业务仓库，也不要混入业务代码目录。

默认统一放置到一个专门的外部工具项目目录中，除非用户当轮明确指定其他路径：

```text
C:\Users\ac\tools-projects
```

执行规则：

- 外部成熟项目的源码 clone、试跑目录、研究用副本，默认放在 `C:\Users\ac\tools-projects` 下。
- 当前业务仓库只保留必要的配置、文档或集成代码；不要把完整外部项目直接复制进业务仓库。
- 临时下载到 `%TEMP%` 的外部项目，如果确认后续还会用，应迁移到 `C:\Users\ac\tools-projects`。
- 工具运行缓存、安装目录、索引数据库可以按工具自身要求放在系统目录，但必须在完成报告里说明实际路径。
- 如果外部项目有 GPL / AGPL / 商业限制许可证，默认只作为参考或单独运行，不直接复制源码进私有业务项目。
- 如果用户有特殊要求，以用户指定路径为准，并在完成报告里写清楚为什么偏离默认目录。
